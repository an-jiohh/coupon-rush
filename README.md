# Coupon Issuance – Limited Stock

선착순 한정 재고 기반의 쿠폰 발급 API 프로젝트

## 요구사항

### 쿠폰 발급 요청 API 구현  

한정된 재고로 선착순으로 재고처리, 누가 뭘 받았는지 알 수 있어야함  
선착순 처리 + 확률 기반 상품 배정  
UID는 path variable로 받아서 처리


- 재고 테이블  
  재고 등록 API는 따로 제공 하지 않고, 서버 구동 시 자동으로 Insert 되도록. 

- 상품
  - 상품 A : 재고 1개, 초기 확률 1%
  - 상품 B : 재고 30개, 초기 확률 10%
  - 상품 C : 재고 69개, 초기 확률 89%(나머지)
  - 중복 발급 불가(UID 단 1회)
  - 실제 추첨 시에는 가용 재고가 있는 상품만 대상으로 확률을 동적으로 재가중치하여 선택

### 동시성 테스트
- 테스트는 200명이 동시에 왔을 때, 처리할 수 있도록 구현
- 테스트 코드 작성, k6 말고 멀티 스레드로 테스트
- 중복 발급 불가

## 동시성 이슈 해결 방안

### 1. 재시도 로직으로 해결(락 X)

1. 사용자 중복 여부 확인
   Unique Insert 고민하였으나, 중복여부를 먼저 판단하면 바로 예외로 발생시키면 다음 로직이 실행되지 않아 이득이라고 판단
2. 쿠폰 후보를 전체 가져와서 확률에 맞추어 뽑음  
3. 선택된 재고를 줄임(조건부 감소)  
   조건부 감소로 남은 요소가 없다면 실패, 실패한다면 다시 시도
4. 재고 차감에 성공했다면 발급 내역을 기록

#### 단점 
- 재시도 로직이 N번 만큼 발생할 수 있음    
  만약 상품이 엄청 많아진다면? 이거를 고려한 설계가 필요함
- 선착순을 보장할 수 없음  
  A와 B가 동시에 같은 쿠폰을 선택해 UPDATE -> 하나가 실패하면 재시도 로직 때문에 후순위로 밀려날 수 있음

임시

실무에서 “선착순”을 보장하는 방법들
1.	DB 단일 자원 제어
- SELECT … FOR UPDATE로 특정 쿠폰 행을 잠그고, 남은 수량 확인 → 감소 → 커밋.
- 동시에 접근한 트랜잭션은 DB 락 순서대로 직렬화됨 → 선착순과 유사한 효과.
- 단점: 락 경합 심할 때 성능 저하.
2.	큐 기반 처리
- API 서버에서는 요청만 큐(Kafka, Redis Stream 등)에 밀어 넣음.
- 별도의 소비자(Consumer)가 단일 스레드로 직렬 처리 → 순서 = 곧 선착순.
- 대규모 이벤트/트래픽에서 많이 쓰이는 방식.
3.	원자적 감소 연산
- Redis DECR 같은 원자 연산 사용.
- 0 이하로 내려가면 실패 → 빠르고 직관적.
- 다만 “분산 환경에서 글로벌 락” 보장하려면 Redis 단일 노드나 Redisson 같은 분산 락 솔루션이 필요.
